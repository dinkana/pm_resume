## Тестовое задание на менеджера проектов 0
Оценить системный подход, навыки планирования, управления рисками и коммуникации.

Компания планирует разработать мобильное приложение для доставки еды с фокусом на здоровое питание. Старт с нуля. У вас есть 2 месяца до публикации минимального рабочего продукта (далее mvp) в сторах.

Команда: 2 мобильных разработчика (iOS/Android), 1 бэкенд-разработчик, 1 дизайнер (на частичной занятости), 1 тестировщик (подключается на 4-й неделе). Технический лид - бэкенд-разработчик.

**Задание:**

1.  На основе описания выше, составьте список из 5-7 ключевых гипотез/рисков для этого проекта. Ранжируйте их по приоритету. Перечислите ключевых стейкхолдеров такого проекта внутри компании. С кем и о чем вам необходимо согласовать требования в первую очередь?
2.  Составьте высокоуровневый список из которого можно создать дорожную карту на 2 месяца, разбив работу на основные этапы. Создайте упрощенный план-график из которого можно создать диаграмму Гантта или таблицу для первого спринта (2 недели), включив ключевые задачи для всех ролей. Используйте условные оценки.
3.  Ситуация 1: Входящее письмо от маркетолога: «Нам нужно добавить в mvp push-уведомления о скидках и кешбэк за первый заказ. Это критично для запуска». Опишите краткий алгоритм ваших шагов (3-5 пунктов) в ответ на это предложение.
    Ситуация 2: За день до конца спринта один из мобильных разработчиков сообщает, что не успевает сделать ключевую задачу по интеграции с api, так как документация со стороны бэкенда была неточной. Опишите краткий план ваших шагов на 3-4 пункта.

## Решение:

### 1. Риски

1.  Неподтвержденная гипотеза о востребованности конкретного предложения "здорового питания" на рынке.
2.  Срыв сроков из-за переоценки скорости разработки mvp за 2 месяца.
3.  Задержки из-за неточных/меняющихся требований к api между бэкендом и мобильными разработчиками.
4.  Низкое качество mvp из-за позднего входа тестировщика на 4-й неделе).
5.  Нехватка ресурсов дизайнера из-за частичной занятости.
6.  Перегрузка/зависимость от единственного бэкенд-разработчика, который также является техлидом.

| **Стейкхолдер** | **Тема для согласования** |
| :--- | :--- |
| **Владелец продукта** | Согласование объема mvp, приоритетов и финальных требований. |
| **Техлид (бэкенд разраб)** | Согласование технической реализуемости, оценок сроков и api. |
| **Маркетинг** | Согласование функциональности, необходимой для запуска и продвижения. |
| **Руководство** | Согласование бюджета, сроков и стратегических целей проекта. |

### 2. Этапы

| **Этап** | **Срок (недели)** | **Содержание** |
| :--- | :--- | :--- |
| **Спринт 0** | 1-2 | Уточнение требований, дизайн, техническое проектирование, настройка окружения. |
| **Спринт 1** | 3-4 | Разработка основных функций (авторизация, каталог, корзина) и базового api. |
| **Спринт 2** | 5-6 | Интеграция frontend-backend, реализация оформления заказа, начало тестов. |
| **Спринт 3** | 7-8 | Исправление багов, приемочное тестирование, подготовка к публикации в сторах. |

**Таблица первого спринта:**

| Роль | Задача | Оценка (дни) |
| :--- | :--- | :--- |
| **Дизайнер** | Доработка и утверждение макетов ключевых экранов (каталог, корзина) | 5 |
| **Бэкенд** | Разработка спецификации api и реализация core-эндпоинтов (каталог, товары) | 10 |
| **Моб. разработчики** | Верстка экранов по готовым макетам, настройка навигации | 10 |
| **ПМ** | Уточнение бэклога, ежедневные стендапы, координация | 10 |

### 3. Работа с требованиями

**Ситуация 1 (запрос от маркетолога):**

1.  Запросить у маркетолога подробное описание требований к приложению.
2.  Запросить у техлида и разработчиков оценку трудозатрат на реализацию.
3.  Оценить влияние на сроки и объем текущего mvp с владельцем продукта.
4.  Принять решение по валидности сроков на требуемый функционал в рамках запланированных спринтов.
5.  Донести решение и его обоснование до маркетолога и команды.

**Ситуация 2 (проблема разработчика перед демо):**

1.  Немедленно подключить техлида (бэкенд) для устранения неясностей в документации api.
2.  Совместно с разработчиком и техлидом оценить, сколько времени потребуется для завершения задачи.
3.  Принять решение: скорректировать план спринта (перенести часть задач), если задержка критична.
4.  Проинформировать команду и стейкхолдеров об изменении плана спринта.
